# Design Iterate — 设计迭代循环

**并行多视角评审 → 整合评估 → [可选] 跨系统 Teams 讨论 → 用户裁决 → 更新原文档**

## 评审产物路径

```
docs/design-reviews/{YYYY-MM-DD}-{批次标识}/
├── 评审计划.md                    # Phase 1 产物（预期文件清单，用于恢复比对）
├── 整合评估.md                    # Phase 3 产物
├── teams-讨论/                    # Phase 3.5 产物（仅在触发 Teams 时）
│   ├── 议题-{N}-{议题标题}.md     # 每个 Teams 讨论议题的完整记录
│   └── teams-综合报告.md          # Teams 讨论综合结论
├── 采纳索引.md                    # Phase 4 产物（采纳项索引，供 parallel-implement 使用）
├── 待定与下轮处理.md              # Phase 4 产物（延后项）
└── {系统名}-{视角名}评审.md       # Phase 2 各 Agent 评审（由 Agent 自行保存）
```

批次标识：单文档→系统名，多文档→批次名（如 `叙事系统批次2`）

---

## Phase 0: 恢复（从已有评审目录继续）

当用户提供一个已有的评审目录路径时，跳过 Phase 1，自动检测进度并继续。

### 检测逻辑

扫描目录内文件，按以下规则判定当前 Phase：

```
目录不存在或为空
  → 未开始，回退到 Phase 1

有 评审计划.md，评审文件不全（比对计划中的预期文件清单）
  → Phase 2 进行中，需补跑缺失的 Agent

有评审文件（*评审.md）齐全，无 整合评估.md
  → Phase 2 完成，从 Phase 3 继续

有 整合评估.md，含跨系统议题但无 teams-讨论/ 目录
  → Phase 3 完成，从 Phase 3.5 继续（如用户选择讨论）

有 整合评估.md + teams-讨论/teams-综合报告.md，无 待定与下轮处理.md
  → Phase 3.5 完成，从 Phase 4 继续

有 整合评估.md，无跨系统议题或用户跳过讨论，无 待定与下轮处理.md
  → Phase 3 完成（无 Teams），从 Phase 4 继续

有 整合评估.md + 采纳索引.md + 待定与下轮处理.md
  → Phase 4 完成，本轮迭代已结束
```

**评审文件完整性检查**：读取 `评审计划.md` 中的预期文件名列表，逐个检查文件是否存在且 >5KB。缺失的文件列出并提示用户是否补跑。若无 `评审计划.md`（旧批次），则跳过完整性检查，仅按文件存在性判断。

### 执行步骤

1. 扫描目录，列出所有文件
2. 按上述规则判定 Phase
3. **向用户确认判定结果**，展示：
   - 目录路径
   - 发现的文件清单
   - 判定的 Phase 和接下来要做什么
4. 用户确认后，跳转到对应 Phase 继续执行
5. 如果从 Phase 4 继续，需读取 `整合评估.md` 中的优化方向清单呈现给用户裁决

## Phase 1: 准备

1. **确定目标文档**
   - 用户指定单个文档 → 直接使用
   - 用户说"全部文档" → **必须分批**（每批 3-5 个，按关联度分组，用户选择顺序）
   - 参数中已含路径 → 直接使用

2. **读取目标文档**，提取版本号、关联文档列表、文档成熟度

3. **确定评审视角**
   - 读取 `references/perspectives/index.md` 获取视角索引和分配策略
   - 根据成熟度和文档类型选择 2-4 个视角
   - **只读取选中视角的文件**（如选了决策+叙事，只读 `06-decision.md` 和 `09-narrative.md`）

4. **检查前置评审** — 在 `docs/design-reviews/` 搜索历史评估，如有则传递给 Agent

5. **向用户确认** — 展示目标文档、版本、选定视角、前次评估，让用户确认或调整

6. **落盘评审计划** — 用户确认后，将计划写入 `docs/design-reviews/{目录}/评审计划.md`，用于 Phase 0 恢复比对

   ```markdown
   # 评审计划

   **批次**: {批次标识}
   **日期**: {YYYY-MM-DD}
   **预期评审文件数**: {N}

   ## 视角分配

   | 文档 | 版本 | 视角 | 预期文件名 |
   |------|------|------|-----------|
   | {文档名} | v{版本} | {视角名} | {系统名}-{视角名}评审.md |
   | ... | ... | ... | ... |
   ```

## Phase 2: 并行评审

**并行启动多个 game-designer Agent**，每个分配独立视角。

### Agent Prompt 模板

```
你是一位专业的游戏设计评审员，专注于【{视角名}】视角。

## 评审目标
评审文档：{文档完整路径}
文档版本：v{版本}

## 你的视角
{从对应视角文件中读取的 prompt 模板}

## 评审要求
1. 仔细阅读目标文档全文
2. 如有关联文档，也需阅读以理解上下文：
   {关联文档路径列表}
3. 如有前次评估，对照检查哪些问题已修复、哪些仍在：
   {前次评估路径（如有）}
4. 从你的视角深度分析，找出 5-15 个发现
5. 每个发现标注严重程度：🔴高 / 🟡中 / 🟢低 / ⭐亮点
6. 提出具体的修复建议（不是泛泛而谈）
7. **跨系统标记**：如果发现涉及其他系统的设计，标注 `[跨系统: {相关系统名}]`

## 输出格式

# {系统名} v{版本} — {视角名}评审

> **评审日期**: {今天日期}
> **评审对象**: `{文档路径}` v{版本}
> **评审视角**: {视角名}

## 发现列表

### {编号} {emoji} {问题标题}
**问题**：{描述}
**分析**：{详细分析}
**建议**：{具体建议}

（重复每个发现）

## 跨系统发现（如有）
- [跨系统: {系统名}] {发现摘要}

## 亮点
- {值得保留的优秀设计}

## 总体评价
{1-2段总结}

## 保存评审文件（必须执行）
评审完成后，你必须使用 Write 工具将完整评审结果保存到以下路径：

`{评审文件保存路径}`

先用 Bash 创建目录（如不存在），再用 Write 写入完整评审内容。
这是你的最后一步，必须在返回结果之前完成。
```

### 执行

- **所有 Agent 并行启动**（同一消息中多个 Task 调用）
- `subagent_type`: `game-designer`
- `description`: `{视角名}评审-{系统名}`

### 文件验证

Agent 返回后，**验证而非保存**：
1. 检查文件是否存在且 >10KB（完整内容的标志）
2. 文件不存在或 <5KB → 用 Agent 返回内容手动补存
3. Phase 3 整合时**从文件读取**，不依赖主对话上下文

## Phase 3: 整合与提议

**必须委托给独立 sub-agent 执行**，不在主对话中进行。这是因为主对话在 Phase 1-2 已积累大量上下文，直接读取所有评审文件会导致上下文溢出。

### 启动整合 Agent

启动一个 `game-designer` agent，`description` 为 `整合评审-{系统名}`。

**Agent Prompt 模板：**

```
你是一位专业的游戏设计整合评审员。你的任务是交叉比对多个视角的评审结果，生成整合评估报告。

## 任务

1. 读取 `{api_reference.md 完整路径}` 获取整合评估输出模板和严重程度标准

2. 读取目录 `{评审产物目录}` 下的所有评审文件（排除整合评估本身）

3. 交叉比对：
   - **共识问题**（2+视角指出）→ 提升优先级
   - **独特发现**（仅1视角）→ 保持原优先级
   - **亮点共识** → 标记"保留"

4. **提取跨系统议题**：
   - 收集所有标注了 `[跨系统]` 的发现
   - 将涉及相同系统组合的发现归类为一个"讨论议题"
   - 每个议题包含：具体问题、涉及的 2-3 份文档、来源评审发现

5. 生成整合评估文档，包含：
   - 多视角共识问题（含各视角的具体发现）
   - 各视角独特发现
   - 优化方向清单（按 P0→P1→P2 排序，每个含：问题、来源视角、建议方案）
   - **跨系统讨论议题清单**（如有）
   - 总体评价

6. 将整合评估保存到 `{评审产物目录}/整合评估.md`

7. **返回精简摘要**（不超过 2000 字），包含：
   - 共识问题数量和标题列表
   - P0/P1/P2 问题数量
   - 优化方向清单（精简版：每条一行，含优先级+标题+来源视角）
   - **跨系统讨论议题**（如有，每条含：议题问题、涉及文档、推荐讨论理由）
   - 指向完整报告的路径

## 跨系统议题输出格式

在整合评估文档中，跨系统议题章节格式：

### 跨系统讨论议题

| # | 议题 | 涉及文档 | 来源发现 | 推荐理由 |
|---|------|---------|---------|---------|
| 1 | {具体问题？} | {文档A} + {文档B} | {评审编号} | {为什么需要跨系统讨论} |
| 2 | ... | ... | ... | ... |

## 评审文件目录
`{评审产物目录}`

## 原始设计文档
`{目标文档路径}` v{版本}
```

### 处理 Agent 返回

Agent 返回的精简摘要会展示给用户。完整报告已保存到磁盘。

1. **验证整合评估文件**已保存
2. **向用户展示**优化方向清单（从 Agent 返回的精简摘要中）
3. **展示跨系统议题**（如有），询问用户：
   - "整合评审发现了 N 个跨系统议题，是否启动 Teams 讨论？"
   - 展示每个议题的问题和涉及文档
   - 用户可以：全部讨论 / 选择部分 / 跳过全部
4. **等待用户讨论** — 呈现清单后等待用户逐条决策，不急于批量更新文档
5. 如用户需要查看某条的详细分析，引导用户查看完整报告路径

**决策分支：**
- 用户选择讨论 → 进入 Phase 3.5
- 用户跳过讨论 → 直接进入 Phase 4
- 用户主动要求讨论某个问题（即使整合 Agent 未标记为跨系统） → 也进入 Phase 3.5

## Phase 3.5: 跨系统 Teams 讨论（问题聚焦模式）

**触发条件（仅在以下情况执行）：**
1. Phase 3 整合发现了跨系统议题，且用户选择讨论
2. 用户主动要求对某个问题进行 Teams 讨论

**核心原则：问题聚焦** — 输入不是"一组文档"，而是"一个具体问题 + 该问题涉及的文档"。

```
❌ "讨论鉴定系统、议价系统和送客系统"（太宽泛）
✅ "合同档位的经济平衡是否需要赎回率绑定？"（具体问题 + 涉及：议价、声誉、经济参数）
```

**执行架构：每个议题独立 Team（视角可定制），串行执行。** 先为所有议题规划视角组合并获得用户确认，再逐个创建 Team 讨论。前序议题结论注入后续议题的成员 prompt，实现跨议题学习。

**详细执行流程**（视角池、Step 0 视角规划、Step 1-7 每议题循环、跨议题上下文注入）见 `references/teams-workflow.md`。

**报告格式**见 `references/api_reference.md` "Teams 综合报告输出格式"。

Teams 讨论结果与 Phase 3 的优化方向清单**合并**后一起进入 Phase 4 裁决。

## Phase 4: 用户裁决与更新

1. **确认采纳清单** — 合并两个来源的建议：
   - Phase 3 的优化方向清单（单文档内部问题）
   - Phase 3.5 的 Teams 讨论结论（跨系统融合方案，如有）

2. **更新原设计文档**（两阶段）
   - **阶段 A: Agent 生成更新版** — 为每个文档启动 game-designer agent，将采纳的变更整合到文档中
     - Agent 写入项目内临时路径：`docs/{系统名}-v{新版本}-updated.md`
     - **注意**：Designer 目录在项目外，Agent 的 Write 工具无法直接写入，必须写到 `docs/` 下
     - **跨系统变更**：如果 Teams 讨论产出涉及多个文档的变更，为每个受影响的文档分别启动 Agent
   - **阶段 B: 主对话复制到 Designer** — Agent 完成后，主对话用 Bash `cp` 命令将临时文件复制到 Designer 目录，然后删除临时文件
   - 版本号递增（v1.1 → v1.2）
   - 更新版本历史，**不删除原有内容**

3. **生成采纳索引** — 写入 `docs/design-reviews/{目录}/采纳索引.md`
   - 索引是**轻量指针**，不复制完整设计内容，只记录"改了什么、在哪里"
   - `parallel-implement` 读索引定位变更点，再读设计文档获取实现细节
   - **格式**：

   ```markdown
   # 采纳索引 — {批次标识}

   > **来源**: `docs/design-reviews/{目录}/整合评估.md`
   > **生成日期**: {YYYY-MM-DD}
   > **文档版本变更**: {文档1} v{旧}→v{新} | {文档2} v{旧}→v{新}

   ## {系统1名}

   **设计文档**: `{Designer/ 路径}` v{旧} → v{新}

   | # | 原编号 | 来源 | 变更摘要 | 设计文档章节 | 代码影响 |
   |---|--------|------|---------|-------------|---------|
   | 1 | P0-1 | 共识(3视角) | 信心条衰减公式修改 | §3.2 | framework |
   | 2 | P1-5 | 视角:体验 | 添加特征发现动画 | §5.1 | interaction |
   | 3 | T-2 | Teams议题1 | 合同档位经济平衡调整 | §4.3 | framework+content |
   | 4 | P2-8 | 视角:叙事 | 补充失败分支文案 | §7.1 | content |
   | 5 | P2-12 | 视角:认知 | 术语统一（"估价"→"评估价"） | §2.1, §3.5 | 无(纯文档) |

   ## {系统2名}
   ...
   ```

   **字段说明**：
   - `原编号`: 对应整合评估中的 P0/P1/P2 编号或 Teams 议题编号
   - `来源`: 共识(N视角) / 视角:{视角名} / Teams议题{N}
   - `代码影响`: `framework` / `interaction` / `content` / `framework+interaction` / `无(纯文档)` — parallel-implement 据此分配任务层
   - `设计文档章节`: 具体章节号，parallel-implement 的 agent 直接定位阅读

   **注意**：如果本轮采纳项全部是纯文档调整（无代码影响），则在索引顶部标注 `**代码影响: 无**`，Phase 5 衔接时跳过 parallel-implement 询问。

4. **记录延后项与保留点子**
   - **延后项**写入 `docs/design-reviews/{目录}/待定与下轮处理.md`
     - 每条包含：原编号、来源共识/视角/Teams、原建议摘要、延后原因、回头处理时的关注点
     - 如有下一批次待评审文档，也列出预览
   - **"拒绝但保留为点子"** 的条目写入 `docs/IDEAS.md`（待评估区），不写入待定文档
     - 包含来源（设计迭代批次X + 原编号/Teams议题）、描述、设计意图、潜在触发条件

5. **输出迭代总结**
   ```
   ## 迭代完成
   **文档**: {文档名}  **版本**: v{旧} → v{新}
   **评估报告**: docs/design-reviews/{目录}/整合评估.md
   **Teams讨论**: docs/design-reviews/{目录}/teams-讨论/（如有）
   **采纳索引**: docs/design-reviews/{目录}/采纳索引.md
   **延后项记录**: docs/design-reviews/{目录}/待定与下轮处理.md
   **本轮变更**: [采纳] N项 (framework: X, interaction: Y, content: Z, 纯文档: W)
   **Teams融合方案**: [采纳] ... / [拒绝] ...
   **待下轮处理**: ...
   ```

6. **衔接代码实现（可选）** — 迭代总结输出后，询问用户：
   > 设计文档已更新。是否启动 `/parallel-implement` 将变更实现到代码？
   - 用户确认 → 调用 `/parallel-implement`，传入本轮评审目录路径
   - 用户拒绝或说"稍后" → 结束本轮迭代
   - 如果采纳索引标注 `**代码影响: 无**`（全部是纯文档调整）→ 跳过此步骤，不询问

## 规则

### 用户主权
- 每个优化方向必须经用户确认，不可自行跳过
- 用户说"全部采纳"时才可批量应用
- 用户拒绝的建议记录到评估文档但不应用
- **讨论优先**：先讨论问题再决定是否采纳，让用户逐条过
- **Teams 讨论是可选的**：用户可以跳过所有跨系统讨论，直接进入 Phase 4

### 分批评审
- 多文档时必须分批（每批 3-5 个），用户选择顺序
- 一批完成后询问是否继续下一批

### Teams 讨论触发条件
- Phase 3 整合 Agent 标记了跨系统议题 + 用户选择讨论
- 用户主动要求（"我想讨论一下 X 和 Y 的关系"）
- **不自动触发** — 始终由用户决定是否启动 Teams
- **视角组合需用户确认** — Teams 讨论前展示每个议题的推荐视角组合（2-4个），用户可调整

### 上下文传递
- 前次评估路径必须传递给 Agent
- Agent 对照前次评估检查修复状态
- 整合时标注"前版已提出但仍未修复"的持续性问题
- Teams 成员接收 Phase 3 的相关发现作为讨论背景

### 文档操作
- 只修改 `Designer/系统设计文档/` 下的设计文档
- 评审产物保存到 `docs/design-reviews/`
- 版本号只增不减

## 上下文韧性

本 skill 涉及大量并行 Agent，主对话上下文可能在等待期间被压缩。以下规则确保流程可恢复：

### Agent 自保存机制
- 每个 game-designer Agent 在 prompt 中被要求**自行保存**评审文件
- 主对话只验证文件存在性，不承担保存职责
- 即使上下文压缩，评审内容已持久化到文件系统

### Phase 3 委托给独立 Agent
- 整合阶段**必须委托给独立 sub-agent 执行**，不在主对话中读取评审文件
- sub-agent 从零开始，拥有完整上下文窗口，从 `docs/design-reviews/` 目录读取评审文件
- 主对话只接收 agent 返回的精简摘要（<2000字），不持有完整评审内容
- 这避免了主对话在 Phase 1-2 积累大量上下文后再读取评审文件导致溢出

### Phase 3.5 Teams 讨论韧性
- **每个议题独立 Team** — 上下文不跨议题累积，消除了连续讨论导致的上下文压缩风险
- 每个议题讨论完毕后立即保存报告到 `teams-讨论/` 目录，然后销毁 Team
- 如果某个议题中断，可通过 Phase 0 恢复：检测已完成的议题报告，从未完成的议题继续
- Teams 成员的消息通过 SendMessage 自动投递，不依赖主对话轮询
- 前序议题结论通过 prompt 注入传递给后续议题，不依赖共享上下文

### 中断恢复
- 如果流程中断（上下文压缩/会话断开），用户可提供评审目录路径触发 **Phase 0 恢复流程**
- Phase 0 自动检测进度并从断点继续，详见上方 Phase 0 章节

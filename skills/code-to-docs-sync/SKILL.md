  触发场景：
  (1) 用户说"同步代码到文档"、"更新设计文档"、"把代码改动同步到文档"
  (2) 用户说"最近的修改还没写到文档里"
  (3) 用户完成一批功能开发后想要同步文档
  (4) 用户说"代码和文档不同步了"

  不适用场景：
  - 纯粹的文档编写（没有对应代码）
  - 代码实现（应使用 programmer）
  - 设计评审（应使用 game-system-review）

# Code to Docs Sync

将代码修改同步到设计文档的工作流程。

## 工作流程

### 阶段 0：检查同步检查点

首先检查是否有上次同步的检查点文件：

```bash
cat .claude/skills/code-to-docs-sync/.sync-checkpoint
```

**如果存在检查点：** 使用增量扫描
```bash
git log --oneline <last_synced_commit>..HEAD | grep -E "^[a-f0-9]+ feat"
```

**如果不存在检查点：** 执行全量扫描（见下方），完成后创建检查点文件。

### 阶段 1：扫描代码修改

**增量扫描（有检查点时）：**
```bash
git log --oneline <last_synced_commit>..HEAD | grep -E "^[a-f0-9]+ feat"
```

**全量扫描（无检查点时）：**
```bash
git log --oneline --all | grep -E "^[a-f0-9]+ feat"
```

获取每个提交修改的文件列表：
```bash
git log --stat --oneline <last_synced_commit>..HEAD
# 或全量：git log -30 --stat --oneline
```

3. **筛选需要同步的修改**（排除以下类型）：
   - `fix(ui):` 纯样式微调（颜色、间距、可见性）
   - `fix:` 小 bug 修复
   - `refactor:` 不改变功能的重构
   - `test:` 测试相关
   - `chore:` 配置、依赖等
   - `.gitignore`、`package.json` 等配置文件变更

4. **识别需要同步的修改**（保留以下类型）：
   - `feat:` 新功能
   - `feat(xxx):` 带模块的新功能
   - 涉及新系统目录的创建（如 `systems/xxx/`）
   - 涉及新组件的创建（如 `components/XxxModal.tsx`）
   - 大量代码变更（>200 行新增）

### 阶段 2：理解代码实现

对每个需要同步的功能：

1. 读取核心代码文件，提取：
   - 数据结构/类型定义
   - 配置常量
   - 主要逻辑流程

2. 整理成文档友好的格式：
   - 功能概述
   - 参数/配置表格
   - 流程说明

### 阶段 3：对比设计文档

1. 定位项目的设计文档目录（从 CLAUDE.md 读取）

2. 检查相关设计文档是否已存在：
   ```bash
   ls <设计文档目录>/系统设计文档/
   ```

3. 读取相关文档，确认哪些内容缺失

### 阶段 4：生成修改摘要（等待用户确认）

向用户展示：

```markdown
## 需要同步到设计文档的修改

### 1. [新功能名称]
**提交:** `xxxxxxx` | **状态:** 设计文档中**未记录**

[功能简述]

| 参数 | 值 | 说明 |
|------|-----|------|
| ... | ... | ... |

---

### 2. [功能名称]
**提交:** `xxxxxxx` | **状态:** 设计文档**需更新**

[变更说明]

---

## 建议的文档更新

**目标文档:** `xxx.md`

需要新增/修改的章节：
1. ...
2. ...

---

**请确认：**
1. 以上修改内容是否准确？
2. 是否同步所有内容，还是只同步部分？
3. 创建独立文档还是整合到现有文档？
```

**关键：必须等待用户确认后才能继续**

### 阶段 5：执行文档同步

根据用户确认：

1. **创建新文档**（如果用户选择独立文档）：
   - 使用标准文档模板
   - 包含版本号、更新日期、状态
   - 完整记录功能设计

2. **更新现有文档**（如果整合到现有文档）：
   - 更新版本号和日期
   - 在适当位置新增章节
   - 添加对新系统的引用链接

3. **更新 CLAUDE.md**（如果有新系统/文档）：
   - Systems Directory 表格
   - Directory Structure
   - Key Documents for Reference

### 阶段 6：输出变更清单

```markdown
**文件变更清单：**
新建: xxx.md
修改: yyy.md
修改: CLAUDE.md
```

## 文档模板

新系统设计文档结构：

```markdown
# [系统名称] ([English Name])

> **版本**: 1.0
> **最后更新**: YYYY-MM-DD
> **状态**: 已实现 - 基于代码同步

---

## 1. 设计初衷

[为什么需要这个系统，解决什么问题]

---

## 2. 系统架构

### 2.1 [核心概念1]
### 2.2 [核心概念2]

---

## 3. [主要功能]详解

### 3.1 概述
### 3.2 配置/参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|

---

## N. 与其他系统的关联

| 关联系统 | 影响 |
|----------|------|

---

## 版本历史

| 版本 | 日期 | 变更 |
|------|------|------|
| 1.0 | YYYY-MM-DD | 初版，基于代码实现同步 |
```

### 阶段 7：更新检查点

同步完成后，更新检查点文件 `.claude/skills/code-to-docs-sync/.sync-checkpoint`：

```
# Code-to-Docs Sync Checkpoint
last_synced_commit: <当前 HEAD hash>
last_sync_date: YYYY-MM-DD
total_commits_at_sync: <总提交数>
feat_commits_at_sync: <feat 提交数>
```

## 注意事项

- 优先读取现有设计文档的风格，保持一致性
- 中文项目使用中文文档，英文项目使用英文文档
- 表格数据直接从代码配置中提取，确保准确性
- 如果代码实现与设计意图有差异，在文档中注明"基于代码同步"
- **每次同步后必须更新检查点文件**

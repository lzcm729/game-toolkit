# 设计文档一致性检查

对多份设计文档做交叉比对，发现文档之间的矛盾和缺失。

## 核心原理

单个 agent 只读自己的文档组，无法发现跨组矛盾。因此采用两阶段法：
- **阶段1（并行提取）**：多个 agent 各自从一组文档中提取结构化数据，不做判断
- **阶段2（交叉比对）**：汇总所有提取结果，比对同一概念在不同文档中的定义是否一致

## 工作流程

### 1. 扫描文档

扫描目标目录，列出所有 `.md` 文件（排除归档目录 `归档/`、`archived/`）。

### 2. 分组

按主题将文档分为 3-5 组，每组 5-10 份。分组原则：
- **相关文档同组**：共享概念的文档放一组，便于发现组内矛盾
- **均衡负载**：每组文档量大致相当
- **边界概念冗余**：跨组的关键概念（如声誉、利率）会在阶段2交叉比对中被捕获

分组前向用户展示分组方案，确认后再启动。

### 3. 阶段1：并行结构化提取

为每组启动一个 Explore agent（`run_in_background: true`），使用提取 prompt 模板（见 `references/extraction-prompt.md`）。

每个 agent 输出六个维度的提取结果：
1. **数值定义** — 所有具体数字（金额、百分比、天数、阈值）
2. **概念定义** — 关键游戏概念的定义
3. **流程与规则** — 状态转换、触发条件、因果关系
4. **枚举列表** — 分类、类型、档位列表
5. **组内矛盾** — 同组文档内发现的不一致
6. **模糊/未定义** — 定义不清、有歧义的地方

每条提取都标注 `[来源文件名]`，便于后续溯源。

### 4. 阶段2：交叉比对

所有 agent 完成后，在主对话中汇总全部提取结果，按以下维度交叉比对：

**比对维度（见 `references/cross-reference-checklist.md`）：**
- 同名概念的不同数值
- 同一操作在不同文档中的声誉影响
- 术语/命名的一致性（是否有改名后的残留）
- 枚举列表的成员是否对齐
- 流程描述是否互相矛盾
- 跨系统接口的参数定义是否匹配

### 5. 生成报告

将发现的问题按严重程度分类：

| 严重程度 | 定义 | 示例 |
|----------|------|------|
| **高** | 同一概念/操作的数值直接矛盾，影响实现 | 文档A说-3，文档B说-1 |
| **中** | 术语不一致、定义分散、接口未对齐 | 改名后旧名残留、参数散落多处无统一来源 |
| **低** | 格式过时、缺失不影响实现的细节 | 枚举格式过时、样例缺初始值 |

报告格式：

```markdown
## 矛盾 N：[简短标题]

**现状：**
| 来源 | 值 |
|------|---|
| 文档A | ... |
| 文档B | ... |

**分析：** [为什么矛盾、哪个更可能是正确的]
**建议：** [具体修正动作]
**严重程度：** 高/中/低
```

### 6. 逐条过审

将报告逐条呈现给用户，每条等待用户确认后再进入下一条。记录用户的裁决（同意/修改/搁置），最终输出确认后的修正清单。

## 输出

最终输出一份**修正清单**，格式：

```markdown
| 编号 | 问题 | 修正动作 | 改哪个文档 | 用户裁决 |
|------|------|---------|-----------|---------|
| M1 | ... | ... | ... | 同意 |
```

## 注意事项

- 只比对正式文档，排除归档目录
- 阶段1的 agent 只提取不判断——判断在阶段2由主对话完成
- 不修改任何文件，只输出报告和修正清单
- 修正清单确认后，用户可选择手动修改或启动其他 agent 执行
